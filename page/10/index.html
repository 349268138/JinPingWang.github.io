IOC, DI, DL

情景分析

Zoo是动物园，它里面有一个依赖，即Mammal引用。从设计模式的角度上，Mammal一般是一个接口，原因是因为哺乳动物可能是老虎，可能是猴子或其它。如果直接将Mammal接口变成Tiger类，则这个就相当于实例化时，只能是Tiger类，如果程序以后想换成其它哺乳动物，如Monkey类，就会报错，因为不能将一个Tiger引用指向Monkey对象。现在有一个问题是，Mammal将要指的对象由谁来创建？假如我们在Zoo的构造函数中这样写（即由Zoo自己来创建）

这样写的一个弊端是，如果以后想让Mammal指向的是Monkey对象呢？那就必需重新修改代码，将new Tiger()修改成new Monkey()，然后重新编译成class文件。可想而知，这样程序的耦合度太大了。好，那我们就换一种写法

现在，通过构造函数的参数来给mammal赋值，且参数mammalPara是接口类型，通过面向对象动态绑定的思想，可知依赖mammal指向的对象并不确定是Tiger, Monkey或是其它哺乳运动。这样程序就很灵活了。现在另一个问题又来了，mammalPara这个引用所指的对象又是由谁来创建的呢？没错，不是在程序中创建的，而是由容器（比如Spring）来创建的。Spring通过在xml中配置一个Mammal接口与一个实体类（如Tiger）的映射，这样程序运行中需要创建一个Mammal对象时，就使用Tiger的构造函数来创建一个对象并赋值给Mammal。而这个映射是写在xml中，所以哪天不想使用Tiger，而要使用Monkey时，则只要将xml中的Tiger改成Monkey即可，源程序无需任何改动。因此程序的耦合性大大降低了。

IOC

inversion fo control, 即控制反转。由上面的情景分析可知，原先由程序自己控制创建对象的权力被容器（Spring）给剥夺了。即创建对象现在反转过来由容器来创建。

DI

dependency injection, 即依赖注入。程序依赖容器创建对象，并将创建的对象注入到程序中。

DL

dependency lookup, 即依赖查找。这个一般用得比较少，

IOC, DI, DL的关系

其实，IOC就是DI+DL，只不过现实中DI相比DL更加适用。所以，平常不撞牛角尖的话，IOC和DI可以认为是默认造价的。

AOP

Aspect Oriented Programing，即面向切面编程。通过横向抽取机制将类无法通过纵向继承体系进行抽象的重复性代码，这些代码一般都是公共的，如性能监控，事务处理这些代码。

JoinPoint

程序执行的某个特定位置。如类某个方法调用前，调用后方法抛出异常后，类开始初始化前，类初始化后。

PointCut

通过切点查找连接点。一个切点相当于数据库查询的条件，可以匹配一个或多个连接点。

Advice

织入到目标类连接点上的一段程序代码。

Target

目标类，即Advice的那段代码会添加到这个类中。

Introduction

特殊的Advice，可以为类添加一些属性和方法。这样即使原始类没有实现某些接口，通过引介就可以动态为该类添加接口的实现逻辑。

Weaving

织入是将Advice添加到Target的过程。有编译期织入，类装载期织入，动态代理织入（Spring采用这一个）

Proxy

将Advice添加到Target后产生的结果类。

Aspect

由切点和Advice组成

SOA

Service-Orientend Architecture，即面向服务的体系结构。

java中的NIO

发表于 2014-01-31   |   分类于 java   |  

IO与NIO的区别

面向块的IO

1：传统的java IO是面向流的IO，流IO一次处理一个字节或一个字符；NIO则是面向块的IO。
2：NIO中引入了缓冲区的概念，缓冲区作为传输数据的基本单位块。传统的java IO也有相应的缓冲区过滤器流(BufferedInputStream等)，但移进移出的操作是由程序员来包装的，而NIO的缓冲区是由底层操作系统来实现的，操作过程如下：某个进程需要进行IO操作，它执行了一次读或写的系统调用，向底层操作系统发出了请求，操作系统会按要求把数据缓冲区填满或排干，这比代码级的实现要高效的多。
3：NIO的缓冲区在数据分析和处理上也带来很大的便利性与灵活性。

java中的流

发表于 2014-01-30   |   分类于 java   |  

java中的流分两种：一种是继承于InputStream, OutputStream抽象类，其中让子类实现的read(), write(int b)都是只读或只写一个字节（0~255）；另一种是继承Reader, Writer抽象类，其中让子类实现的read(char cbuf[], int off, int len), void write(char cbuf[], int off, int len)只读或只写一个字符（java中的一个字符包含两个字节）。这两种流使用时不应该交叉使用，即用OutputStream写，就不提倡使用Reader读，使用Writer写，就不提倡使用InputStream读，因为有可能会出错。此外，InputStream, OutputStream, Reader, Writer等都实现了Closeable接口，所以jdk1.7后的版本都可以使用带资源的try块语句。
